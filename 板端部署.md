# CTC_FRDM 板端部署方案

## 1. 项目定位

本方案用于指导 `CTC` 项目在 `CTC_FRDM` 目录下完成板端部署设计。

当前 `CTC` 项目已经完成了 PC 端训练闭环，模型输入是**帧提取后的骨架时序序列**，不是原始 RGB 视频。因此，板端系统的核心任务不是直接做视频分类，而是要围绕以下链路展开：

```text
摄像头视频流
  -> 逐帧关键点提取
  -> 骨架时序序列构建
  -> 连续词边界检测
  -> 孤立词分类推理
  -> gloss 序列输出
  -> 大模型 API 翻译成自然语言
```

也就是说，板端系统本质上是一个**基于骨架时序序列的实时连续手语识别与翻译系统**。

---

## 2. 当前基础能力

PC 端 `CTC` 项目已经具备以下基础：

- 已完成孤立词训练流程
- 已完成骨架序列数据整理
- 已完成标签映射整理
- 已完成模型导出代码补充
- 已确认当前训练主线是 `single` 骨架分类模型
- 数据集已完成 500 词规模的全量训练准备

因此，板端部署不需要从零构建，而是要复用现有骨架时序训练体系。

---

## 3. 你的系统思路

你的核心思路是：

1. 通过摄像头采集实时画面
2. 提取骨架关键点
3. 根据关键帧变化和双手下垂状态判断词边界
4. 将切分后的骨架片段送入孤立词模型
5. 输出手语 `gloss` 序列
6. 再通过大模型 API 将 `gloss` 转成自然语言

这个思路是合理的，而且非常适合当前项目阶段。

---

## 4. 为什么要这样设计

### 4.1 你当前训练的是骨架序列
当前模型训练输入不是视频像素，而是：

- 每帧骨架关键点
- 组成的时序序列

所以板端如果直接拿视频做分类，会和训练时的数据分布不一致。

### 4.2 骨架序列更适合做实时系统
和原始视频相比，骨架序列具有以下优点：

- 输入更轻量
- 计算更稳定
- 更利于边界检测
- 更容易和孤立词模型对齐
- 更适合板端实时运行

### 4.3 更适合分阶段落地
这个方案可以先做：

- 摄像头 -> 骨架提取 -> 孤立词推理

然后再做：

- 连续边界检测 -> gloss 拼接

最后再接：

- 大模型翻译

这样最稳。

---

## 5. 目标硬件与部署现实

### 5.1 目标板卡
- NXP i.MX93
- Arm Ethos-U65 Micro-NPU

### 5.2 现实约束
Ethos-U65 更适合：

- 卷积类算子
- 量化后的轻量模型
- 规则明确、图结构简单的推理链路

而 `BiLSTM` 这类循环结构一般不适合作为 NPU 主力算子，因此：

- PC 端可以保留当前训练基线
- 板端应优先考虑更轻量、更 NPU 友好的结构
- 若暂时无法完全上 NPU，可先 CPU 跑通，再逐步优化

---

## 6. 总体系统架构

推荐采用分层架构，而不是端到端单模型硬解。

```text
[Camera]
   |
   v
[Frame Capture]
   |
   v
[Landmark Extraction]
   |
   v
[Skeleton Sequence Buffer]
   |
   v
[Gesture Boundary Detection]
   |
   v
[Isolated Word Inference]
   |
   v
[Gloss Sequence Builder]
   |
   v
[LLM API Translation]
   |
   v
[Final Natural Language Output]
```

## 7. 关键模块设计

### 7.1 摄像头采集
负责获取实时画面帧。

要求：
- 低延迟
- 稳定帧率
- 易与后续骨架提取模块衔接

---

### 7.2 骨架关键点提取
这一层是你当前训练数据的来源，也是板端系统的核心预处理。

处理内容：
- 从每帧图像中提取上半身关键点
- 提取左手关键点
- 提取右手关键点
- 拼接成单帧骨架特征向量

输出：
- 实时骨架序列缓存

---

### 7.3 骨架序列缓冲区
将连续帧骨架数据保存为滑动缓存或事件缓存。

建议：
- 维护一个短时帧缓存
- 支持按词边界切片
- 支持固定长度重采样

---

### 7.4 边界检测
这是连续识别的关键。

你的想法是：

> 通过关键帧变化，根据双手放下的状态来判断一个手语词的结束。

这个思路可以落地为状态机。

#### 状态机建议
- `IDLE`：未检测到有效手语动作
- `ACTIVE`：检测到动作开始，持续缓存骨架序列
- `ENDING`：检测到动作逐渐收束，准备切词
- `EMIT`：输出一个词片段并回到空闲状态

#### 边界条件建议
一个词结束可以由以下条件共同决定：
- 双手速度下降到阈值以下
- 双手位置进入下垂区域
- 连续若干帧变化率显著降低
- 片段持续时长达到最小阈值

---

### 7.5 孤立词识别
这一层复用当前 PC 端训练模型。

输入：
- 切分后的骨架时序片段

输出：
- 某个手语词的类别 ID
- 对应的 `gloss`
- 置信度

说明：
- 当前训练模型是孤立词分类模型
- 因此板端应尽量把连续流切成一个个词片段再识别
- 不建议一开始就直接做端到端连续识别

---

### 7.6 gloss 序列构建
将每次识别结果追加到序列缓存中，形成连续 `gloss`。

建议加入：
- 重复结果抑制
- 低置信度丢弃
- 相邻片段合并
- 时间间隔去抖

输出示例：

```text
[我, 想, 去, 学校, 学习]
```

---

### 7.7 大模型 API 翻译
`gloss` 序列通常不是自然语言最终语序，因此需要大模型 API 做语义重写。

作用：
- 调整语序
- 补全省略信息
- 转换为自然语言句子

输入示例：
```text
gloss: 我 想 去 学校 学习
```

输出示例：
```text
我想去学校学习。
```

---

## 8. 推荐实施路线

### Phase 1：板端骨架序列闭环
目标：
- 摄像头采集
- 骨架提取
- 固定长度序列整理
- 孤立词模型推理

这一阶段先不做复杂连续翻译，只验证基础链路。

---

### Phase 2：连续边界检测
目标：
- 识别动作开始与结束
- 根据手部下垂和关键帧变化切分词片段

这一阶段的重点是把“一个词”切准。

---

### Phase 3：gloss 序列输出
目标：
- 把多个词片段按顺序拼起来
- 输出连续 `gloss` 序列

---

### Phase 4：LLM 自然语言翻译
目标：
- 将 gloss 序列转换成自然语言文本
- 完成最终翻译输出

---

## 9. 关于模型结构的建议

### 当前 PC 端
当前你可以继续保留：
- `single` 模型
- `TCNBiLSTM` 训练基线

### 板端候选
板端更建议逐步演进到：
- 更轻量的 `TCN` / `CNN` 风格模型
- 更少的循环结构依赖
- 更容易量化和编译的网络

### 原则
- PC 端优先保证精度
- 板端优先保证实时性和可部署性
- 如果需要，后续再把 PC 端模型压缩/替换为板端友好模型

---

## 10. 风险点

### 10.1 边界检测不稳
如果双手放下的动作不明显，容易切错词。

应对方式：
- 使用多条件判定
- 设置最小时长
- 加入去抖机制

### 10.2 词间过渡动作复杂
连续手语里词与词之间常有过渡动作，可能导致误切分。

应对方式：
- 先做规则版
- 再采集连续数据改进边界检测

### 10.3 板端算力受限
i.MX93 的算力和 NPU 适合轻量模型。

应对方式：
- 先 CPU 跑通
- 再逐步优化到 NPU 友好结构

### 10.4 LLM API 延迟
如果翻译请求发送到云端，可能引入延迟。

应对方式：
- 缓存 gloss
- 批量发送句子片段
- 根据场景决定是否实时调用

---

## 11. 建议的工程拆分

建议在 `CTC_FRDM` 目录下拆成以下功能块：

- `capture/`：摄像头采集
- `landmark/`：骨架提取
- `buffer/`：骨架缓存和切片
- `boundary/`：边界检测
- `inference/`：孤立词推理
- `gloss/`：序列拼接与去抖
- `translation/`：LLM API 调用
- `ui/`：结果显示

---

## 12. 你这个方案的核心价值

这个方案的优点是：

- 和当前训练数据完全对齐
- 不需要一开始就做复杂端到端连续识别
- 更容易做出可运行的板端系统
- 更适合 i.MX93 + Ethos-U65 的现实条件
- 便于后续逐步升级为更强的连续翻译系统

---

## 13. 结论

当前项目最合理的方向是：

1. 继续把 PC 端全量 500 词训练跑好
2. 将训练模型导出
3. 在板端先完成**摄像头 -> 骨架序列 -> 孤立词推理**闭环
4. 再加入**边界检测**做连续识别
5. 输出 `gloss` 序列
6. 最后用大模型 API 翻译成自然语言

这条路线最符合你当前的数据形态、模型形态和板端硬件条件。
